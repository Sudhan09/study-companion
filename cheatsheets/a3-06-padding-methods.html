6 · Padding methods

Make a string fit a fixed width by adding filler. Strings only grow — they never shrink.
Loop Week · Day 3 · Block A.3 · Subtopic 6

zfill — pad left with zeros

"42".zfill(5)            →  "00042"        (3 zeros + "42" = 5 chars)
"7".zfill(3)             →  "007"
"-12".zfill(5)           →  "-0012"        (handles the sign correctly)

# Already big enough? No change.
"hello".zfill(3)         →  "hello"
Used for · invoice numbers, file indexing (img_001.jpg), time formatting (05:09), zero-padded IDs.

center / ljust / rjust — direction-based padding

s = "hi"

s.center(10)             →  "    hi    "       (default fill = space)
s.center(10, "*")        →  "****hi****"       (custom fill)

s.ljust(8)               →  "hi      "         (push left, pad right)
s.ljust(8, "-")          →  "hi------"

s.rjust(5)               →  "   hi"            (push right, pad left)
s.rjust(5, "0")          →  "000hi"            (same as zfill on the result)
Mnemonic · ljust pushes the content to the left (so padding goes on the right). rjust pushes the content to the right. The name says where the content lands, not where the padding goes.

The no-shrink rule

# If the string is already that wide or wider, you get the original back
"hello".rjust(3)         →  "hello"        (5 chars, asked for 3 — no change)
"hello".center(5)        →  "hello"        (already 5 wide)
"hello".zfill(2)         →  "hello"        (already > 2 chars)

# Padding only ADDS chars. It never trims.
Lock · width arg is a minimum, not a maximum. To enforce a maximum, slice the result: name.ljust(10)[:10].

f-string equivalents (cleaner in most cases)

# These pairs do the same thing:
"42".zfill(5)            ≡   f"{42:05d}"          →  "00042"
"hi".center(10)          ≡   f"{'hi':^10}"        →  "    hi    "
"hi".ljust(10)           ≡   f"{'hi':<10}"        →  "hi        "
"hi".rjust(10)           ≡   f"{'hi':>10}"        →  "        hi"
"42".rjust(5, "0")       ≡   f"{42:0>5}"          →  "00042"
Which to use? Methods are clearer when you have a string already in hand. f-strings are clearer when you're building a string from values. Both are fine.

Real-world patterns

# 1. Zero-padded sequence (filenames, IDs)
for i in range(1, 11):
    name = f"image_{str(i).zfill(3)}.jpg"        # image_001.jpg, image_002.jpg, ...

# 2. Aligned table column
for row in data:
    print(row['name'].ljust(20), row['score'].rjust(6))

# 3. Banner / divider
print("DONE".center(40, "="))                    # "==================DONE=================="

# 4. Time formatting (manual)
def pretty_time(h, m, s):
    return f"{str(h).zfill(2)}:{str(m).zfill(2)}:{str(s).zfill(2)}"

Lock these

  1. .zfill(n) pads with zeros on the left. Handles signs correctly.
  2. .ljust / .rjust / .center — name tells you where the content goes.
  3. No-shrink rule · the width is a minimum, not a maximum. Strings only grow.
  4. All these have f-string equivalents — use whichever reads cleaner for your case.
  5. Default fill is a space. Pass a 2nd arg for a custom fill character.
